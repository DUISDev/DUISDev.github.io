## Основные понятия

| Термин | Описание |
|--------|----------|
| **Рабочая директория** | Файлы на диске, с которыми вы работаете прямо сейчас |
| **Staging (индекс)** | Промежуточная область: сюда попадают изменения перед коммитом (`git add`) |
| **Коммит** | Снимок состояния проекта с сообщением, автором и датой |
| **Ветка (branch)** | Указатель на цепочку коммитов |
| **HEAD** | Текущая позиция в истории (обычно — последний коммит текущей ветки) |
| **Remote** | Удалённый репозиторий (GitHub, GitLab и т.д.) |

Типичный цикл:

```
изменили файлы → git add → git commit → git push
```

## Настройка и инициализация

| Команда | Назначение |
|---------|------------|
| `git init` | Создать новый локальный репозиторий в текущей папке |
| `git clone <url>` | Скопировать удалённый репозиторий на компьютер |
| `git config user.name "Имя"` | Задать имя автора коммитов (локально) |
| `git config user.email "email@example.com"` | Задать email автора коммитов |
| `git config --global ...` | Те же настройки, но для всех репозиториев пользователя |
| `git config --list` | Показать текущие настройки |

Пример первоначальной настройки:

```bash
git config --global user.name "Иван Иванов"
git config --global user.email "ivan@example.com"
```

## Ежедневные команды

| Команда | Назначение |
|---------|------------|
| `git status` | Показать состояние: изменённые, новые и проиндексированные файлы |
| `git add <файл>` | Добавить конкретный файл в staging |
| `git add .` | Добавить все изменения в текущей папке и подпапках |
| `git add -p` | Интерактивно выбрать части изменений (hunks) для staging |
| `git commit -m "сообщение"` | Создать коммит с указанным сообщением |
| `git commit -am "сообщение"` | Добавить в коммит **только уже отслеживаемые** файлы и сразу закоммитить |
| `git diff` | Показать не проиндексированные изменения |
| `git diff --staged` | Показать изменения, уже добавленные в staging |
| `git restore <файл>` | Отменить изменения в рабочей директории (до последнего коммита) |
| `git restore --staged <файл>` | Убрать файл из staging, не трогая рабочую копию |
| `git rm <файл>` | Удалить файл и зафиксировать удаление |
| `git mv старый новый` | Переименовать или переместить файл с фиксацией в Git |

## Работа с ветками

| Команда | Назначение |
|---------|------------|
| `git branch` | Список локальных веток |
| `git branch <имя>` | Создать новую ветку |
| `git checkout <ветка>` | Переключиться на другую ветку |
| `git switch <ветка>` | Современная альтернатива `checkout` для переключения веток |
| `git switch -c <имя>` | Создать ветку и сразу переключиться на неё |
| `git merge <ветка>` | Влить указанную ветку в текущую |
| `git rebase <ветка>` | Перенести коммиты текущей ветки поверх другой |
| `git branch -d <имя>` | Удалить ветку (если она уже слита) |
| `git branch -D <имя>` | Принудительно удалить ветку |

## Удалённые репозитории

| Команда | Назначение |
|---------|------------|
| `git remote -v` | Список удалённых репозиториев и их URL |
| `git remote add origin <url>` | Привязать удалённый репозиторий с именем `origin` |
| `git fetch` | Загрузить изменения с remote, **не** меняя локальные файлы |
| `git pull` | `fetch` + слияние с текущей веткой |
| `git push` | Отправить локальные коммиты на remote |
| `git push -u origin <ветка>` | Первый push ветки и привязка к upstream |
| `git push origin --delete <ветка>` | Удалить ветку на remote |

## Просмотр истории и откат изменений

| Команда | Назначение |
|---------|------------|
| `git log` | История коммитов |
| `git log --oneline --graph` | Компактная история с графом веток |
| `git show <хеш>` | Детали конкретного коммита |
| `git reset --soft HEAD~1` | Отменить последний коммит, **оставив изменения в staging** |
| `git reset --mixed HEAD~1` | Отменить коммит, изменения останутся в рабочей директории (не в staging) |
| `git reset --hard HEAD~1` | Отменить коммит и **удалить все его изменения** (осторожно!) |
| `git revert <хеш>` | Создать новый коммит, отменяющий указанный (без переписывания истории) |
| `git stash` | Временно спрятать незакоммиченные изменения |
| `git stash pop` | Вернуть спрятанные изменения |

## Гайд: как сделать коммит

### Шаг 1. Проверить состояние

```bash
git status
```

Убедитесь, что видите нужные файлы в разделах *Changes not staged* или *Untracked files*.

### Шаг 2. Добавить файлы в staging

```bash
# один файл
git add src/app.js

# все изменения
git add .

# только часть изменений в файле
git add -p
```

### Шаг 3. Проверить, что попадёт в коммит

```bash
git diff --staged
```

### Шаг 4. Создать коммит

```bash
git commit -m "Краткое описание: что и зачем изменено"
```

**Хорошее сообщение коммита:**
- в повелительном наклонении: «Добавить авторизацию», а не «Добавил авторизацию»;
- одна логическая задача — один коммит;
- первая строка — до ~50–72 символов; при необходимости добавьте пустую строку и подробное описание.

Пример с многострочным сообщением:

```bash
git commit -m "$(cat <<'EOF'
Исправить падение при пустом списке пользователей

Добавлена проверка на пустой массив перед рендером таблицы.
EOF
)"
```

### Шаг 5. Отправить на remote (если нужно)

```bash
git push
# или при первом push ветки:
git push -u origin main
```

### Полный пример

```bash
git status
git add .
git diff --staged
git commit -m "Обновить README: добавить инструкцию по установке"
git push
```

## Как полностью удалить коммит

«Полностью удалить» может означать разное — выберите подходящий сценарий.

### Сценарий A: коммит ещё **не отправлен** на remote (`git push` не делали)

#### A1. Удалить последний коммит, сохранив изменения в файлах

```bash
# изменения останутся проиндексированными (в staging)
git reset --soft HEAD~1

# изменения останутся в файлах, но не в staging (по умолчанию)
git reset HEAD~1
```

#### A2. Удалить последний коммит **и все его изменения**

```bash
git reset --hard HEAD~1
```

> **Внимание:** `--hard` безвозвратно удаляет изменения из рабочей директории.

#### A3. Удалить несколько последних коммитов

```bash
# удалить последние 3 коммита, сохранив изменения в файлах
git reset --soft HEAD~3

# удалить последние 3 коммита вместе с изменениями
git reset --hard HEAD~3
```

#### A4. Удалить коммит не с конца истории (переписать историю)

```bash
# интерактивный rebase: замените N на число коммитов «назад» от HEAD
git rebase -i HEAD~N
```

В открывшемся редакторе:
- удалите строку с ненужным коммитом, **или**
- замените `pick` на `drop` для этого коммита.

Сохраните и закройте редактор.

---

### Сценарий B: коммит **уже отправлен** на remote

Если другие люди могли уже скачать этот коммит, **не переписывайте историю** без согласования. Безопасный способ:

```bash
git revert <хеш_коммита>
git push
```

Это создаст **новый** коммит, который отменяет изменения старого. История остаётся линейной и предсказуемой.

#### Если коммит нужно именно стереть из истории (force push)

Используйте только если вы уверены, что никто не опирается на эту историю:

```bash
# локально удалить последний коммит
git reset --hard HEAD~1

# перезаписать remote (опасно!)
git push --force-with-lease
```

`--force-with-lease` безопаснее обычного `--force`: push не пройдёт, если на remote появились новые коммиты, которых у вас нет.

---

### Сценарий C: удалить коммит, но оставить файлы как были **до** него

```bash
git reset --hard HEAD~1
```

Или откатиться к конкретному коммиту:

```bash
git reset --hard <хеш_нужного_коммита>
```

---

### Сценарий D: «удалить» коммит из истории, но сохранить его изменения как новый коммит

```bash
git revert <хеш>
```

Или через rebase — «склеить» (squash) несколько коммитов в один:

```bash
git rebase -i HEAD~3
# замените pick на squash (s) у коммитов, которые нужно объединить
```

---

### Сводная таблица

| Ситуация | Команда | Изменения в файлах |
|----------|---------|-------------------|
| Отменить последний коммит, сохранить в staging | `git reset --soft HEAD~1` | Сохраняются |
| Отменить коммит, оставить в рабочей папке | `git reset HEAD~1` | Сохраняются |
| Полностью убрать коммит и изменения | `git reset --hard HEAD~1` | **Удаляются** |
| Отменить уже запушенный коммит (безопасно) | `git revert <хеш>` | Создаётся обратный коммит |
| Переписать историю на remote | `git reset --hard ...` + `git push --force-with-lease` | Зависит от reset |

## GitHub

Git — система контроля версий. **GitHub** — облачный сервис для хранения Git-репозиториев и совместной работы: веб-интерфейс, pull request'ы, issues, CI/CD и т.д. Аналогичные платформы — GitLab, Bitbucket; принципы ниже в большинстве случаев те же.

### Основные сущности GitHub

| Сущность | Описание |
|----------|----------|
| **Repository (repo)** | Проект на GitHub: код, история коммитов, ветки, issues |
| **Owner** | Пользователь или организация, которой принадлежит репозиторий |
| **Fork** | Ваша копия чужого репозитория в вашем аккаунте |
| **Clone** | Локальная копия репозитория на компьютере |
| **Branch** | Ветка в репозитории (на GitHub отображается так же, как в Git) |
| **Pull Request (PR)** | Запрос на слияние изменений из одной ветки в другую; место для обсуждения и ревью |
| **Issue** | Задача, баг или предложение; можно привязать к PR |
| **Release** | Стабильная версия проекта, часто с тегом и архивом исходников |
| **Actions** | Встроенный CI/CD: автоматические проверки и деплой по событиям (push, PR и др.) |

### Создание и клонирование репозитория

**На GitHub (через сайт):**
1. Кнопка **New repository** → имя, видимость (public/private), опционально README и `.gitignore`.
2. После создания GitHub покажет команды для подключения.

**Локально — новый проект → GitHub:**

```bash
git init
git add .
git commit -m "Initial commit"
git branch -M main
git remote add origin https://github.com/USER/REPO.git
git push -u origin main
```

**Клонирование существующего репозитория:**

```bash
# HTTPS
git clone https://github.com/USER/REPO.git

# SSH
git clone git@github.com:USER/REPO.git
```

| Способ | URL | Аутентификация |
|--------|-----|----------------|
| **HTTPS** | `https://github.com/USER/REPO.git` | Логин + [Personal Access Token (PAT)](https://github.com/settings/tokens) (пароль аккаунта для Git не используется) |
| **SSH** | `git@github.com:USER/REPO.git` | SSH-ключ, добавленный в [настройках GitHub](https://github.com/settings/keys) |

Генерация SSH-ключа (если ещё нет):

```bash
ssh-keygen -t ed25519 -C "your_email@example.com"
cat ~/.ssh/id_ed25519.pub   # скопировать в GitHub → Settings → SSH keys
ssh -T git@github.com       # проверка подключения
```

### Типовые сценарии работы

#### Прямой доступ к репозиторию (вы — collaborator или владелец)

```bash
git switch -c feature/my-change    # новая ветка
# ... правки ...
git add .
git commit -m "Описание изменений"
git push -u origin feature/my-change
```

На GitHub: **Compare & pull request** → создать PR → после ревью **Merge**.

#### Fork + Pull Request (вклад в чужой проект)

1. На странице репозитория — **Fork**.
2. Клонировать **свой** fork, не оригинал:

```bash
git clone https://github.com/YOUR_USER/REPO.git
cd REPO
git remote add upstream https://github.com/ORIGINAL_USER/REPO.git
```

3. Перед новой задачей синхронизироваться с upstream:

```bash
git fetch upstream
git switch main
git merge upstream/main
git push origin main
```

4. Ветка → коммиты → push в **свой** fork → PR из fork в **upstream**.

### Pull Request: что это и как устроен

PR — не команда Git, а функция GitHub. Вы предлагаете: «влейте мои коммиты из ветки A в ветку B».

**Типичный цикл PR:**

1. Создать ветку и запушить её.
2. На GitHub: **Pull requests → New pull request**.
3. Выбрать **base** (куда вливаем, обычно `main`) и **compare** (откуда, ваша ветка).
4. Заполнить заголовок и описание: что сделано, зачем, как проверить.
5. Дождаться ревью и CI (если настроен).
6. **Merge** одним из способов:
   - **Merge commit** — сохраняет все коммиты ветки как есть;
   - **Squash and merge** — все коммиты ветки сжимаются в один;
   - **Rebase and merge** — коммиты переносятся линейно поверх base.

После merge ветку-источник обычно удаляют (кнопка на GitHub или локально):

```bash
git switch main
git pull
git branch -d feature/my-change
git push origin --delete feature/my-change   # удалить на remote
```

### Issues

Issues — трекер задач внутри репозитория.

| Действие | Где |
|----------|-----|
| Создать issue | **Issues → New issue** |
| Привязать к PR | В описании PR: `Fixes #42` или `Closes #42` — при merge PR issue закроется автоматически |
| Метки, milestone, assignee | Фильтрация и планирование на странице issue |

### Releases и теги

Теги в Git отмечают конкретный коммит (часто версию). Release на GitHub — тег + описание + опционально бинарные файлы.

```bash
git tag -a v1.0.0 -m "Release 1.0.0"
git push origin v1.0.0
```

На GitHub: **Releases → Draft a new release** → выбрать тег, описать изменения.

### GitHub Actions (кратко)

Файлы workflow в `.github/workflows/*.yml` задают автоматизацию: тесты, линтеры, сборка, деплой.

Типичные триггеры:
- `push` — при каждом push;
- `pull_request` — при открытии/обновлении PR;
- `schedule` — по расписанию (cron).

Статус проверок виден в PR (зелёная/красная галочка). Без прохождения CI merge иногда блокируется правилами репозитория (**branch protection**).

### Права и настройки репозитория

| Роль / настройка | Возможности |
|------------------|-------------|
| **Read** | Клонирование, просмотр |
| **Write** | Push в репозиторий (если не включена защита веток) |
| **Maintain / Admin** | Настройки, правила, удаление репо |
| **Branch protection** | Требовать PR, ревью, прохождение CI перед merge в `main` |
| **CODEOWNERS** | Автоназначение ревьюеров по путям файлов (`.github/CODEOWNERS`) |

### Команда `gh` (GitHub CLI)

Официальная утилита для работы с GitHub из терминала. Установка: [cli.github.com](https://cli.github.com/).

```bash
gh auth login                  # авторизация
gh repo clone USER/REPO        # клонировать
gh repo create                 # создать репозиторий
gh pr create                   # создать PR из текущей ветки
gh pr list                     # список PR
gh pr checkout 123             # переключиться на ветку PR #123
gh issue list                  # список issues
gh release create v1.0.0       # создать release
```

`gh` необязателен — всё то же можно сделать через сайт и обычный `git`.

### Частые операции на GitHub

| Задача | Git / GitHub |
|--------|--------------|
| Обновить локальную `main` | `git pull origin main` |
| Посмотреть remote | `git remote -v` |
| Сменить URL remote | `git remote set-url origin <новый-url>` |
| Открыть репозиторий в браузере | `gh repo view --web` или URL на github.com |
| Скачать архив без Git | **Code → Download ZIP** на странице репо |
| Посмотреть diff в PR | Вкладка **Files changed** |
| Обсудить строку кода | **Review → Add single comment** или **Start a review** |

### Безопасность на GitHub

1. **Не храните секреты в коде** — API-ключи, пароли, токены. Используйте [GitHub Secrets](https://docs.github.com/en/actions/security-for-github-actions/security-guides/using-secrets-in-github-actions) для Actions и переменные окружения локально.
2. **PAT** — выдавайте минимально необходимые права и срок действия.
3. **Dependabot / security alerts** — GitHub предупреждает об уязвимых зависимостях (если включено).
4. Если секрет случайно попал в коммит — считайте его скомпрометированным: отзовите ключ и удалите из истории (сложная операция; лучше ротация секрета).

### GitHub vs Git: что где делать

| Действие | Инструмент |
|----------|------------|
| Коммиты, ветки, merge локально | Git |
| Хранение репозитория, PR, issues, CI | GitHub |
| `git push` / `git pull` | Мост между локальным Git и GitHub |
| Code review, обсуждение | GitHub (в PR) |
| Откат коммита локально | Git (`reset`, `revert`) |
| Защита главной ветки, правила merge | GitHub (настройки репозитория) |

## Полезные команды

| Команда | Назначение |
|---------|------------|
| `git blame <файл>` | Кто и когда менял каждую строку |
| `git tag v1.0.0` | Поставить метку на текущий коммит |
| `git tag -a v1.0.0 -m "Релиз 1.0"` | Аннотированный тег |
| `git cherry-pick <хеш>` | Перенести один коммит в текущую ветку |
| `git clean -fd` | Удалить неотслеживаемые файлы и папки |
| `git reflog` | Журнал всех перемещений HEAD (спасает после `--hard`) |
| `git fsck` | Проверка целостности репозитория |

### Восстановление «потерянного» коммита через reflog

Если случайно сделали `reset --hard`:

```bash
git reflog
# найдите хеш нужного состояния, например abc1234
git reset --hard abc1234
```

## Частые ошибки и советы

1. **Перед `--hard` и `--force`** — убедитесь, что важные изменения сохранены или запушены в другую ветку.
2. **Не коммитьте секреты** — `.env`, ключи, пароли. Добавьте их в `.gitignore`.
3. **Делайте `git pull` перед `git push`**, если работаете в общей ветке.
4. **Используйте ветки** для задач и фич, а не коммитьте всё в `main`.
5. **`.gitignore`** — файл со списком того, что Git должен игнорировать (сборки, зависимости, локальные конфиги).
6. При конфликте слияния Git помечает файлы маркерами `<<<<<<<`, `=======`, `>>>>>>>`. Исправьте вручную, затем:

```bash
git add <файл_с_конфликтом>
git commit
```

## Шпаргалка: минимальный набор для ежедневной работы

```bash
git status              # что изменилось
git add .               # подготовить к коммиту
git commit -m "..."     # зафиксировать
git pull                # получить чужие изменения
git push                # отправить свои
git log --oneline -10   # последние 10 коммитов
git branch              # список веток
git switch -c feature   # новая ветка для задачи
```
