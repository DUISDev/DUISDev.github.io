Гайд: как сделать коммит

Шаг 1. Проверить состояние

git status

Убедитесь, что видите нужные файлы в разделах Changes not staged или Untracked files.

Шаг 2. Добавить файлы в staging

# один файл
git add src/app.js

# все изменения
git add .

# только часть изменений в файле
git add -p

Шаг 3. Проверить, что попадёт в коммит

git diff --staged

Шаг 4. Создать коммит

git commit -m "Краткое описание: что и зачем изменено"

Хорошее сообщение коммита:

  • в повелительном наклонении: «Добавить авторизацию», а не «Добавил авторизацию»;
  • одна логическая задача — один коммит;
  • первая строка — до ~50–72 символов; при необходимости добавьте пустую строку и подробное описание.

Пример с многострочным сообщением:

git commit -m "$(cat <<'EOF'
Исправить падение при пустом списке пользователей

Добавлена проверка на пустой массив перед рендером таблицы.
EOF
)"

Шаг 5. Отправить на remote (если нужно)

git push
# или при первом push ветки:
git push -u origin main

Полный пример

git status
git add .
git diff --staged
git commit -m "Обновить README: добавить инструкцию по установке"
git push

Как полностью удалить коммит

«Полностью удалить» может означать разное — выберите подходящий сценарий.

Сценарий A: коммит ещё не отправлен на remote (git push не делали)

A1. Удалить последний коммит, сохранив изменения в файлах

# изменения останутся проиндексированными (в staging)
git reset --soft HEAD~1

# изменения останутся в файлах, но не в staging (по умолчанию)
git reset HEAD~1

A2. Удалить последний коммит и все его изменения

git reset --hard HEAD~1
Внимание: --hard безвозвратно удаляет изменения из рабочей директории.

A3. Удалить несколько последних коммитов

# удалить последние 3 коммита, сохранив изменения в файлах
git reset --soft HEAD~3

# удалить последние 3 коммита вместе с изменениями
git reset --hard HEAD~3

A4. Удалить коммит не с конца истории (переписать историю)

# интерактивный rebase: замените N на число коммитов «назад» от HEAD
git rebase -i HEAD~N

В открывшемся редакторе:

  • удалите строку с ненужным коммитом, или
  • замените pick на drop для этого коммита.

Сохраните и закройте редактор.

Сценарий B: коммит уже отправлен на remote

Если другие люди могли уже скачать этот коммит, не переписывайте историю без согласования. Безопасный способ:

git revert <хеш_коммита>
git push

Это создаст новый коммит, который отменяет изменения старого. История остаётся линейной и предсказуемой.

Если коммит нужно именно стереть из истории (force push)

Используйте только если вы уверены, что никто не опирается на эту историю:

# локально удалить последний коммит
git reset --hard HEAD~1

# перезаписать remote (опасно!)
git push --force-with-lease

--force-with-lease безопаснее обычного --force: push не пройдёт, если на remote появились новые коммиты, которых у вас нет.

Сценарий C: удалить коммит, но оставить файлы как были до него

git reset --hard HEAD~1

Или откатиться к конкретному коммиту:

git reset --hard <хеш_нужного_коммита>

Сценарий D: «удалить» коммит из истории, но сохранить его изменения как новый коммит

git revert <хеш>

Или через rebase — «склеить» (squash) несколько коммитов в один:

git rebase -i HEAD~3
# замените pick на squash (s) у коммитов, которые нужно объединить

GitHub

Git — система контроля версий. GitHub — облачный сервис для хранения Git-репозиториев и совместной работы: веб-интерфейс, pull request'ы, issues, CI/CD и т.д. Аналогичные платформы — GitLab, Bitbucket; принципы ниже в большинстве случаев те же.

Создание и клонирование репозитория

На GitHub (через сайт):

  1. Кнопка New repository → имя, видимость (public/private), опционально README и .gitignore.
  2. После создания GitHub покажет команды для подключения.

Локально — новый проект → GitHub:

git init
git add .
git commit -m "Initial commit"
git branch -M main
git remote add origin https://github.com/USER/REPO.git
git push -u origin main

Клонирование существующего репозитория:

# HTTPS
git clone https://github.com/USER/REPO.git

# SSH
git clone git@github.com:USER/REPO.git

Типовые сценарии работы

Прямой доступ к репозиторию (вы — collaborator или владелец)

git switch -c feature/my-change    # новая ветка
# ... правки ...
git add .
git commit -m "Описание изменений"
git push -u origin feature/my-change

На GitHub: Compare & pull request → создать PR → после ревью Merge.

Fork + Pull Request (вклад в чужой проект)

  1. На странице репозитория — Fork.
  2. Клонировать свой fork, не оригинал:
git clone https://github.com/YOUR_USER/REPO.git
cd REPO
git remote add upstream https://github.com/ORIGINAL_USER/REPO.git
  1. Перед новой задачей синхронизироваться с upstream:
git fetch upstream
git switch main
git merge upstream/main
git push origin main
  1. Ветка → коммиты → push в свой fork → PR из fork в upstream.

Pull Request: что это и как устроен

PR — не команда Git, а функция GitHub. Вы предлагаете: «влейте мои коммиты из ветки A в ветку B».

Типичный цикл PR:

  1. Создать ветку и запушить её.
  2. На GitHub: Pull requests → New pull request.
  3. Выбрать base (куда вливаем, обычно main) и compare (откуда, ваша ветка).
  4. Заполнить заголовок и описание: что сделано, зачем, как проверить.
  5. Дождаться ревью и CI (если настроен).
  6. Merge одним из способов:
  • Merge commit — сохраняет все коммиты ветки как есть;
  • Squash and merge — все коммиты ветки сжимаются в один;
  • Rebase and merge — коммиты переносятся линейно поверх base.

После merge ветку-источник обычно удаляют (кнопка на GitHub или локально):

git switch main
git pull
git branch -d feature/my-change
git push origin --delete feature/my-change   # удалить на remote

Releases и теги

Теги в Git отмечают конкретный коммит (часто версию). Release на GitHub — тег + описание + опционально бинарные файлы.

git tag -a v1.0.0 -m "Release 1.0.0"
git push origin v1.0.0

На GitHub: Releases → Draft a new release → выбрать тег, описать изменения.

GitHub Actions (кратко)

Файлы workflow в .github/workflows/*.yml задают автоматизацию: тесты, линтеры, сборка, деплой.

Типичные триггеры:

  • push — при каждом push;
  • pull_request — при открытии/обновлении PR;
  • schedule — по расписанию (cron).

Статус проверок виден в PR (зелёная/красная галочка). Без прохождения CI merge иногда блокируется правилами репозитория (branch protection).

Команда gh (GitHub CLI)

Официальная утилита для работы с GitHub из терминала. Установка: cli.github.com.

gh auth login                  # авторизация
gh repo clone USER/REPO        # клонировать
gh repo create                 # создать репозиторий
gh pr create                   # создать PR из текущей ветки
gh pr list                     # список PR
gh pr checkout 123             # переключиться на ветку PR #123
gh issue list                  # список issues
gh release create v1.0.0       # создать release

gh необязателен — всё то же можно сделать через сайт и обычный git.

Безопасность на GitHub

  1. Не храните секреты в коде — API-ключи, пароли, токены. Используйте GitHub Secrets для Actions и переменные окружения локально.
  2. PAT — выдавайте минимально необходимые права и срок действия.
  3. Dependabot / security alerts — GitHub предупреждает об уязвимых зависимостях (если включено).
  4. Если секрет случайно попал в коммит — считайте его скомпрометированным: отзовите ключ и удалите из истории (сложная операция; лучше ротация секрета).

Частые ошибки и советы

  1. Перед --hard и --force — убедитесь, что важные изменения сохранены или запушены в другую ветку.
  2. Не коммитьте секреты — .env, ключи, пароли. Добавьте их в .gitignore.
  3. Делайте git pull перед git push, если работаете в общей ветке.
  4. Используйте ветки для задач и фич, а не коммитьте всё в main.
  5. .gitignore — файл со списком того, что Git должен игнорировать (сборки, зависимости, локальные конфиги).
  6. При конфликте слияния Git помечает файлы маркерами <<<<<<<, =======, >>>>>>>. Исправьте вручную, затем:
git add <файл_с_конфликтом>
git commit
